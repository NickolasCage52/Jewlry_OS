# Demo-to-Web Alignment Guide

Операционное руководство для команд **Design**, **Front-end** и **QA**. Документ встраивает принципы ТЗ «приведение web к уровню demo-эталона» в ежедневную работу репозитория.

## 1. Эталон и приоритет источников

1. **Demo** (корень проекта: `styles/*.css`, `src/components/*`, `src/modules/*`) — **единственный утверждённый эталон** визуала, композиции, плотности, иерархии, поведения и UX.
2. **Утверждённая спецификация выравнивания** — документы в `docs/ui-governance/`, заполненная матрица расхождений, зафиксированные токены после Stage 2+.
3. **Макеты** (если появятся) — не отменяют demo при конфликте.
4. **Локальные решения команды** — допустимы только если не противоречат пунктам 1–3.

## 2. Допустимые отклонения от demo

Отклонение **допустимо только** при **документировании** в матрице (колонки exception / rationale) и наличии основания:

1. Объективные ограничения web-среды.
2. Адаптивность **без** разрушения иерархии и сценария.
3. Доказуемое улучшение usability (критерий проверки указан).
4. Требования accessibility **без** деградации характера интерфейса эталона.

Иное — **дефект качества**.

## 3. Пять слоёв работы (последовательность)

| Слой | Содержание |
|------|------------|
| **1. Governance** | Документы в `docs/ui-governance/`, gates, DoD (текущий репозиторий). |
| **2. Audit** | Матрица demo ↔ web по экранам и состояниям; фиксация расхождений. |
| **3. Design system alignment** | Токены из `styles/variables.css` → единый слой для `web/` (Stage 2). |
| **4. Screen & flow alignment** | Страницы и потоки до паритета (Stage 2+). |
| **5. Quality gates** | Регрессия, чеклисты, блокировка релиза при деградации. |

**Запрещено** на ранних этапах: хаотичные локальные правки страниц без обновления токенов и матрицы.

## 4. Обязательные домены сравнения

При аудите и приёмке сверять (см. также чеклисты):

структура экранов, layout, композиция, сетка, отступы, размеры, пропорции, иерархия, типографика, цвет, иконки, карточки, кнопки, формы, таблицы, фильтры, меню, модалки, навигация; состояния: hover, active, focus-visible, disabled, loading, error, empty, no access, long content; микроанимации; плотность; когнитивная нагрузка; скорость считывания первичной информации; сценарии; общее ощущение целостности.

## 5. Логика строки матрицы

Для **каждого** экрана и значимого состояния:

`demo (ссылка/модуль) → web (route) → расхождение → действие | исключение + rationale`.

Строка без решения не считается закрытой для релиза целевого экрана.

## 6. Роли

- **Design:** заполняет эталон demo, правила адаптива, обновляет матрицу при изменении эталона.
- **Front-end:** реализует паритет; любое отклонение — только через exception в матрице.
- **QA:** верифицирует матрицу, gates, субъективный критерий «web не слабее demo».
