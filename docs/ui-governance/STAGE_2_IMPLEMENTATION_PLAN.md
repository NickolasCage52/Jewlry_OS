# Stage 2 — Implementation Plan (после Governance + Audit Framework)

Stage 1 **не меняет UI**. Stage 2 — **системное выравнивание** demo → web. Ниже порядок работ, согласованный с приоритетом из мастер-промпта проекта.

## Цели Stage 2

1. Единый **источник дизайн-токенов** из эталона `styles/variables.css` (и связанных правил) в приложении `web/`.
2. Выравнивание **App shell** и **навигации** до паритета с demo.
3. Выравнивание **ядра**: Dashboard, Sales, Content Center, Catalog (в пределах наличия эталона в demo).
4. Выравнивание **общих примитивов**: кнопки, поля, карточки, вкладки, модалки, таблицы, dropdown, фильтры — через общую папку компонентов (например `web/components/ui/`).

## Фазы

### Фаза 2A — Токены и базовая тема (блокирующая)

- Экспорт/дублирование переменных из demo в `web` (один `:root` / `@theme` источник, без второй палитры).
- Подключение шрифтов **как в demo** (Cormorant Garamond + Jost) или документированное exception.
- Базовые стили body, фон, текст — паритет с `styles/global.css`.

**Критерий готовности:** Gate F на «глобальные стили»; скрин shell до/после.

### Фаза 2B — App shell + navigation

- `AppShell.tsx`: ширина сайдбара, топбар, отступы main, активные пункты — по `styles/components.css` / demo JS.
- Мобильное поведение: явные правила vs demo `@media`.

**Критерий:** строки `W1-SHELL`, `W1-NAV` в матрице → `parity`.

### Фаза 2C — Страницы Wave 1

Порядок:

1. `/app/dashboard`
2. `/app/sales`, `/app/sales/[id]`
3. `/app/content`, `/app/content/[id]`
4. `/app/catalog`, `/app/catalog/[sku]` (при отсутствии полного demo — `n_a` + консистентность с токенами и shell)

**Критерий:** матрица обновлена; Gate Q для каждого маршрута.

### Фаза 2D — Примитивы и состояния

- Вынести повторяющиеся паттерны в UI-библиотеку внутри репозитория.
- Реализовать состояния из матрицы для примитивов до паритета с `Modal.js` и паттернами форм в demo.

### Фаза 2E — Визуальная регрессия (расширение)

- По [VISUAL_REGRESSION_MINIMUM.md](./VISUAL_REGRESSION_MINIMUM.md): при стабилизации — Playwright screenshot tests на 3–5 маршрутах.

## Зависимости и риски

- **Demo имеет только 3 крупных модуля** — остальные страницы web потребуют либо расширения demo, либо статуса `n_a` с внутренними стандартами.
- **Данные:** для честной сверки нужны сопоставимые mock-данные (в web уже есть `lib/mock/*` — проверить паритет с `src/data/*`).

## Выход Stage 2

- Закрытая Wave 1 матрица (или большинство строк `parity`).
- Единая токен-система в web.
- Документированный следующий scope (Wave 2): operations, procurement, channels, и т.д.
